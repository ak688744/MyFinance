# L1.5 — Unified Investment Model + First-Class Accounts — Design Spec

**Date:** 2026-06-14
**Layer:** L1.5 (between L1 Ingestion and L2 Web UI)
**Tier:** T1 (new layer + reshapes `core` schema → **Groww re-validation required**)
**Status:** Design approved; ready for `writing-plans`.
**Inputs:** [frontend design spike](2026-06-13-frontend-design-spike-design.md) §6 (schema payload), §3–§5 (UI contract); `MASTER_PLAN.md` §4 L1.5 row.

---

## 0. What this is

Generalize the **MF-only** investment foundation into an **all-asset-class** model so the
investment section becomes a single **net-worth view** across mutual funds, fixed deposits,
PPF/EPF/NPS, gold, real estate, cash — plus **first-class liabilities** (loans) netted out,
and **first-class accounts** for both investments and expenses.

This is the layer the frontend design spike was run to de-risk: the mockups proved which
data dimensions the platform must carry **before** the schema is poured. This spec turns
that payload into a concrete `core` design.

### Terminal state
This spec is the contract. It proceeds to `writing-plans` (T1), then a branch
`layer/1.5-unified-investment`, TDD build, subagent code review, PR, MASTER_PLAN update.

## 1. Locked scope decisions (from brainstorm, 2026-06-14)

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D1 | **v1 class scope** | Implement **MF** (market, existing pipeline), **manual** (gold, real estate, cash), **computed** (FD, PPF, EPF, NPS), **loans** (amortization). **Defer** live market pricing for **stocks / gold-ETF** (needs a new price provider). | Every valuation strategy (market/computed/manual) is exercised without a new live-price integration. `assetClass` enum stays **open** so deferred classes are drop-in later. |
| D2 | **MF data path** | **Keep MF tables as-is; `portfolio.ts` XIRR math runs UNCHANGED.** New classes live in new tables; the net-worth rollup **unions** the MF path with the new classes. | Lowest re-validation risk — the Groww gate becomes "MF pipeline untouched + golden-master still green", not "rewrite aggregation and re-prove to 0.046pp". |
| D3 | **Global assets surface** | **Read-time unified projection (Option 2).** Physical `assets` table holds only Family 2/3 classes. A domain-level `getAllAssets()` projects MF holdings + `assets` rows into **one uniform list** for rollup/API/UI. | MF value/units/XIRR are **always derived**, never stored — materializing MF rows in `assets` would duplicate the source of truth and add staleness + import-sync burden for zero benefit. The projection gives one logical surface with none of that. |
| D4 | **Net-worth history** | **Derive-on-read** from cashflows + price/value history (no net-worth snapshot table). Current net worth is the **must-have**; the historical **trend series** is this layer's **stretch goal**. | Always exact, no snapshot job. MF history needs **historical NAV** (mfapi.in serves it; `NavLookup.getNAVForDate` already exists). Manual assets get a **valuation-history table** (`asset_valuations`) so a past value is well-defined; computed assets are deterministic by formula at any date. |
| D5 | **New financial math validation** | **Textbook-formula unit tests** (EMI/amortization, compound-interest/FD-maturity). No external oracle for these (deterministic closed forms). XIRR continues to be guarded by the Groww golden-master. | These formulas are textbook-deterministic, unlike XIRR convergence. Real-statement reconciliation (day-count/rounding) is a noted later item, not a v1 blocker. |
| D6 | **Rate over time** | **Rate schedule** (`asset_rates`: rate periods per asset). The valuation engine compounds each contribution under whichever rate applied in each period. FD = single period. | PPF/EPF rates are reset periodically by the govt; a single constant rate drifts over a long horizon. Accurate long-horizon answer, cheap to add now. |
| D7 | **Contribution shape** | **`asset_contributions` stream** table. FD = 1 row (the principal); PPF/EPF/NPS = N rows (recurring deposits); manual (gold/RE/cash) = 0 rows. | FD and PPF are *not* the same shape: FD is a one-time deposit, PPF/EPF/NPS receive ongoing contributions to the same account. The stream is also exactly what derive-on-read history consumes. |
| D8 | **Accounts** | **First-class, scoped per-domain** (`investment` / `expense`), as a **two-level entity** `(institution, label)`. Includes **expense accounts** in this layer (`transactions` gains nullable `account_id`). | Per-domain matches how users think and the existing `(accountName, investmentApp)` columns (spike §5.4 — deliberate divergence from the old "one global accounts" note). Two-level = **multiple Groww accounts** (same institution, distinct labels) is a first-class case. |
| D9 | **AI insight slots** | **Deferred to L4.** Not modeled in `core` now. | The L4 agent layer will own insight generation/storage; reserving a half-designed seam now is speculative. |
| D10 | **Budgets** | **Deferred to L2.** Not in the `core` reshape. | Budgets are an expense-reads/UI concern; L2 decides storage against the concrete Expenses layout. |
| D11 | **Migration** | **No data migration** — there is no production data yet. Pure **schema evolution**: new tables + one nullable column on `transactions`. Accounts are created naturally going forward (MF import upserts an account; manual forms create directly). | Drops the migration/backfill ceremony entirely. The Groww gate is the standing re-validation **invariant**, not a data-preservation check. |

**Three structural families** (the organizing insight):

| Family | Classes | Shape | Storage | Valuation |
|---|---|---|---|---|
| **1 — Transactional / market-priced** | MF (now); **stocks (deferred)** | transaction stream, units/qty × live price, **XIRR** | existing `investment_*` tables (untouched); stocks later **extend this pipeline, NOT `assets`** | `market` — existing `portfolio.ts` |
| **2 — Single-value** | FD; gold, real estate, cash | one deposit (FD) or a stated number | `assets` (+ `asset_contributions` 1 row for FD; `asset_valuations` for manual) | `computed` (FD) / `manual` (gold/RE/cash) |
| **3 — Contributory / computed** | PPF, EPF, NPS | recurring contributions, **no units/NAV**, rate may change | `assets` + `asset_contributions` (N) + `asset_rates` | `computed` |

> **Stocks note (important for future readers):** stocks belong to **Family 1**, not the
> generic `assets` table. They are structurally MF with relabeled fields
> (`scheme→instrument`, `units→quantity`, `NAV→LTP`, `PURCHASE/REDEMPTION→BUY/SELL`) and
> will be implemented by **extending the MF pipeline** (generalize `investment_schemes`
> → `instruments`, or a sibling table behind the same repo seam) when live pricing is
> added. Do **not** try to store stocks in `assets`.

## 2. Schema

MF tables (`investment_schemes`, `investment_transactions`) and all `portfolio.ts` /
`returns.ts` logic are **untouched**. Money/rate/value columns stay **REAL** (consistent with
the L0 money decision). All tables use CHECK/UNIQUE/INDEX constraints in the existing style.

### 2.1 New tables (Drizzle, `packages/core/src/db/schema.ts`)

```
accounts
  id              INTEGER PK autoincrement
  domain          TEXT NOT NULL  CHECK in ('investment','expense')
  asset_class     TEXT NULL                       -- optional finer scope (e.g. 'ppf')
  institution     TEXT NOT NULL                   -- 'Groww','Zerodha','SBI','HDFC Bank'
  label           TEXT NOT NULL                   -- 'Groww — Personal','Groww — Spouse'
  created_at      TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  UNIQUE(domain, institution, label)

assets                                            -- Family 2 & 3 ONLY (never MF/stocks)
  id                 INTEGER PK autoincrement
  account_id         INTEGER NOT NULL  FK -> accounts.id
  asset_class        TEXT NOT NULL  CHECK in ('ppf','epf','nps','fd','gold','real_estate','cash')
  name               TEXT NOT NULL
  valuation_strategy TEXT NOT NULL  CHECK in ('computed','manual')   -- 'market' is MF-only
  ingestion_mode     TEXT NOT NULL  CHECK in ('manual_entry','file_import') DEFAULT 'manual_entry'
  params             TEXT NULL                      -- JSON: compounding_freq, maturity_date, payout, grams, ...
  status             TEXT NOT NULL  CHECK in ('active','closed') DEFAULT 'active'
  opened_at          TEXT NULL
  created_at         TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  updated_at         TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  INDEX(account_id), INDEX(asset_class)

asset_contributions                                -- money-in stream
  id                 INTEGER PK autoincrement
  asset_id           INTEGER NOT NULL  FK -> assets.id
  contribution_date  TEXT NOT NULL
  amount             REAL NOT NULL
  note               TEXT NULL
  created_at         TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  INDEX(asset_id, contribution_date)

asset_rates                                        -- rate schedule (rate periods)
  id                 INTEGER PK autoincrement
  asset_id           INTEGER NOT NULL  FK -> assets.id
  effective_from     TEXT NOT NULL
  rate               REAL NOT NULL                  -- annual %, e.g. 7.1
  created_at         TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  UNIQUE(asset_id, effective_from)
  INDEX(asset_id, effective_from)

asset_valuations                                   -- manual stated-value history
  id                 INTEGER PK autoincrement
  asset_id           INTEGER NOT NULL  FK -> assets.id
  value              REAL NOT NULL
  valued_at          TEXT NOT NULL
  note               TEXT NULL
  created_at         TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  INDEX(asset_id, valued_at)

liabilities                                        -- loans (first-class)
  id              INTEGER PK autoincrement
  account_id      INTEGER NULL  FK -> accounts.id
  name            TEXT NOT NULL
  loan_type       TEXT NOT NULL  CHECK in ('home','car','personal','other')
  principal       REAL NOT NULL
  annual_rate     REAL NOT NULL                     -- annual %
  tenure_months   INTEGER NULL                      -- nullable if emi_amount supplied instead
  emi_amount      REAL NULL                         -- nullable if derived from tenure
  start_date      TEXT NOT NULL
  status          TEXT NOT NULL  CHECK in ('active','closed') DEFAULT 'active'
  created_at      TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  updated_at      TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
  CHECK (tenure_months IS NOT NULL OR emi_amount IS NOT NULL)
```

### 2.2 Modified table

```
transactions  (expenses)
  + account_id  INTEGER NULL  FK -> accounts.id     -- nullable; existing/legacy rows = implicit default
```

`investment_transactions` / `investment_schemes`: **no column or data change.**

### 2.3 `params` JSON shapes (validated in the domain layer, never queried in SQL)

| Class | `params` fields |
|---|---|
| `fd` | `compounding` (`monthly`/`quarterly`/`half_yearly`/`yearly`), `maturity_date`, `payout` (`cumulative`/`periodic`) |
| `ppf`/`epf`/`nps` | `compounding` (default `yearly`), optional `maturity_date` |
| `gold` | `grams` (informational; value is `manual`) |
| `real_estate` / `cash` | none required (value is `manual`) |

JSON holds **only** the heterogeneous per-class inputs the valuation engine consumes; all
shared, queryable columns are explicit. `params` is validated with a per-class schema (Zod
or hand-written guard) in the domain layer, not the DB.

## 3. Valuation engine — `packages/core/src/domain/valuation/`

Pure, repo-injected (depends on interfaces, not Drizzle), `today` injectable for determinism.
Uniform output type so all strategies are interchangeable in the projection.

```ts
type ValuedAsset = {
  assetId: number | null;          // null for MF projections that key off scheme/account
  assetClass: string;
  accountId: number | null;
  name: string;
  valuationStrategy: 'market' | 'computed' | 'manual';
  currentValue: number;
  invested: number | null;         // Σ contributions (computed); null for pure manual
  returns: number | null;          // currentValue − invested; null if invested null
  asOf: string;                    // date the value is computed for
  valuedAt?: string;               // manual: when the user last stated it
  ageDays?: number;                // manual: today − valuedAt (freshness chip)
};
```

- **`valueComputedAsset(asset, contributions, rates, today)`** — for each contribution,
  compound `contribution_date → today` using the rate in effect per `asset_rates` period
  (compounding frequency from `params`). `invested = Σ amounts`, `currentValue = Σ compounded`,
  `returns = currentValue − invested`. FD = single contribution + single rate period (value
  respects `maturity_date`/payout). PPF/EPF/NPS = N contributions, possibly multi-period.
- **`valueManualAsset(asset, valuations, today)`** — latest `asset_valuations` by `valued_at`;
  `{ currentValue, valuedAt, ageDays }`, `invested/returns = null` unless contributions exist.
- **`market`** — NOT reimplemented; MF stays in `portfolio.ts`. The engine only defines the
  shared `ValuedAsset` shape so MF results sit beside computed/manual ones.

**Tests (textbook oracles, D5):** compound-interest & FD-maturity closed forms; a hand-computed
multi-contribution + rate-change schedule; manual freshness/age.

## 4. Liabilities / amortization — `packages/core/src/domain/loans/`

Pure, `today` injectable.

- **`amortizationSchedule(loan)`** — reducing-balance EMI. EMI from `principal`,
  `annual_rate/12`, `tenure_months` (standard formula); if `emi_amount` given, use it and
  derive tenure. Returns `{ period, dueDate, emi, principalComponent, interestComponent, balance }[]`.
- **`loanStatus(loan, today)`** — `{ outstanding, paidPrincipal, interestPaid,
  interestRemaining, monthsRemaining, nextDueDate, progressPercent }`.
- **Net-worth contribution:** a loan adds **−outstanding**.

**Tests (textbook oracles):** known EMI for a sample loan; hand-verified short schedule
(principal/interest split each month, final balance → 0); interest-remaining sum.
**Known later item:** day-count / rounding conventions vs a real bank statement (not a v1 blocker).

## 5. Net-worth rollup + unified projection — `packages/core/src/domain/networth/`

- **`getAllAssets(deps, { account?, assetClass? }, today) → ValuedAsset[]`** — the unified
  projection (D3): MF holdings (via existing `getHoldings`/`portfolio.ts`, mapped to
  `ValuedAsset`, `assetClass='mutual_fund'`, `strategy='market'`) **unioned** with Family-2/3
  assets (via the valuation engine). The single logical "global assets" surface.
- **`getNetWorth(deps, { account? }, today) → NetWorthSummary`:**
  ```
  totalAssets      = Σ getAllAssets().currentValue
  byAssetClass     = [{ assetClass, value, percentage, count }]   // composition bar + KPI cards
  totalLiabilities = Σ loanStatus().outstanding
  netWorth         = totalAssets − totalLiabilities
  ```
- **`getNetWorthHistory(deps, { range }, today) → { date, netWorth, byAssetClass }[]`**
  (**stretch**, D4): for each sampled past date `d` — MF value (units-held-as-of-`d` ×
  **NAV-at-`d`** via `NavLookup.getNAVForDate`), computed assets compounded to `d`, manual
  assets = latest valuation **as of `d`**, liabilities = outstanding at `d`. Summed.
  - **Dependency flagged:** needs historical NAV per scheme. The `NavLookup` interface
    **already exposes `getNAVForDate(amfiCode, date)`** (L0 nav service implements it), so no
    interface extension is required — but wiring/caching full per-scheme history across many
    sample dates may be heavy → the history endpoint may ship **scoped/optional** in this
    layer. **Current net worth is the must-have; the trend series is the stretch.**

## 6. Repository + API surface

### 6.1 New repos (own all SQL, interface-injected — preserves the L0 seam)
- `AccountRepo` — `list({domain?})`, `create`, `findByTriple(domain, institution, label)`,
  `ensureAccount(...)` (upsert used by MF import).
- `AssetRepo` — CRUD; `list({account?, assetClass?})`, `getById`.
- `AssetContributionRepo`, `AssetRateRepo`, `AssetValuationRepo` — `insert`, `listByAsset`.
- `LiabilityRepo` — CRUD; `list({status?})`, `getById`.
- Extend `ExpenseTransactionRepo` to carry `account_id` (insert + reads).

### 6.2 New / extended API routes (`packages/api`, L1 patterns; `app.inject()` tests)
- `GET /networth` → rollup summary. `GET /networth/history?range=` → trend (stretch).
- `GET /assets` (`?account=`, `?assetClass=`) → unified projection. `GET /assets/:id`.
- `POST /assets` (manual-entry CRUD: creates asset + contributions + rates/valuation per class);
  `PATCH /assets/:id`; `DELETE /assets/:id`.
- `POST /assets/:id/contributions`, `POST /assets/:id/valuations`, `POST /assets/:id/rates`.
- `GET /accounts` (`?domain=`), `POST /accounts`.
- `GET /liabilities`, `GET /liabilities/:id` (incl. amortization schedule),
  `POST /liabilities`, `PATCH /liabilities/:id`, `DELETE /liabilities/:id`.
- Expense reads/import extended to carry `account_id` (**non-financial**).
- MF import path calls `ensureAccount` (**non-financial** sync).

`buildServer` shape unchanged (injectable `amfiMatch`, `NavLookup`, txRunner).

## 7. Testing strategy & the T1 gate

**Primary core-logic gate (T1, D2):**
- The existing **Groww golden-master** (`packages/core/test/golden/groww.golden.test.ts`,
  6 schemes / 104 txns, ≤0.5pp; SCHEME_D ≤0.001pp) and the **full existing core suite** pass
  **unchanged** — proving the schema reshape didn't couple into the MF pipeline.
- **Rollup continuity:** `getNetWorth` over MF-only data (no Family-2/3 assets, no liabilities)
  == existing `getPortfolioSummary` total, exactly.

**New-math tests (textbook oracles, D5):**
- Valuation engine: compound-interest / FD-maturity closed forms; multi-contribution +
  rate-change schedule; manual freshness/age.
- Amortization: known EMI; hand-verified schedule (principal/interest split, balance → 0);
  interest-remaining.

**Structural / integration:**
- Net worth: assets − liabilities; per-class composition; history sample points (NAV-for-date
  **stub**, offline & deterministic).
- Repos: fake-repo unit tests for domain; better-sqlite3-backed repo tests.
- API: `app.inject()` route tests, offline (injected `amfiMatch`/NAV stubs).
- **Seam invariant:** zero Drizzle/better-sqlite3 imports anywhere in `domain/`.

**Process:** TDD throughout (T1 non-negotiable). Build via `superpowers:subagent-driven-development`
(fresh subagent per task), branch `layer/1.5-unified-investment` off `main`; final
`feature-dev:code-reviewer` pass before PR. Node 20 (`source ~/.nvm/nvm.sh && nvm use 20`).
Typecheck via `tsc --build` (api → core project ref; build core first). Push/PR via
`gh auth switch --user ak688744`.

## 8. Out of scope (explicitly deferred)

- **Live market pricing** for stocks / gold-ETF (new price provider) — later layer. `assetClass`
  enum stays open; stocks extend the **MF pipeline**, not `assets`.
- **AI insight slots** in `core` (D9) → L4.
- **Budgets** in `core` (D10) → L2.
- **Enriched expense reads** (date-range/direction/search filters, `/expenses/summary`,
  category breakdowns) → L2 (already deferred from L1).
- **Real-statement reconciliation** of loan/FD math (day-count/rounding) → later, when a
  concrete statement is available.
- **Net-worth history** may ship scoped/optional within this layer (D4) — current net worth is
  the must-have.
- **Multi-currency / multi-member** — out (single-user, INR).
